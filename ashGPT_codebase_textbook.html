The ashGPT Codebase, Explained End to End

A guided tour of every part of this multi node LangGraph agent. Architecture, libraries, retrieval pipeline, prompts, evaluation, ablations, and worked examples that show the system reasoning over Australian property law.

Welcome

ashGPT is an experimental retrieval augmented agent that helps a law student study Australian property law. It is built around a single research question. Can a multi node agent that pulls chronological facts apart from binding legal rules produce more accurate causal analyses than a single prompt summariser, when both see the exact same retrieved evidence. The repository sets up the pipeline, the data, the layered retrieval stack, the post synthesis verification node, the evaluation harness, and the frontend needed to answer that question with measurable evidence.

Design hypothesis. Cognitive separation versus attention dilution. A multi node LangGraph state machine that separates ratio decidendi extraction from chronological fact extraction will produce more accurate causal analyses than a single consolidated mega prompt baseline that receives identical retrieved context.

The system is intentionally compact. Six cognitive nodes including a post synthesis verification stage, one ChromaDB collection with metadata filters, a cross encoder reranker that re scores the diverse MMR pool, three model providers behind a single dispatcher, and a twenty two case evaluation suite that scores four configurations head to head. Every architectural choice in the project flows from the hypothesis. The router exists so that ratio and chronology can be measured independently. The dual index exists so that lecture slides count as evidence on the same footing as PDF readings. The reranker exists to lift the most relevant chunks above the diversity driven candidates. The verification node exists to catch any case citation that survived synthesis but is not anchored in the retrieved chunks. The two stage cross provider judge exists so that the model that wrote the answer never grades it alone.

What changed since the first draft

The codebase has grown beyond the original five node design. The new pieces are all observable in the repository.

  • A cross encoder reranker sits after MMR. src/agent/reranker.py wraps cross-encoder/ms-marco-MiniLM-L-6-v2. tools.py over fetches with MMR and reduces with the cross encoder.
  • A verification node runs after synthesis. src/agent/verification.py extracts every plausible case citation from the final answer and checks it against the retrieved chunk text. nodes.py hosts the node itself. The compiled graph wires it after synthesis when USE_VERIFICATION is on.
  • The eval harness now compares four configurations. Plain LLM Baseline, Mega Prompt Ablation, Agent without reranker, and the Full Agent. The fourth column isolates the reranker's contribution.
  • The eval suite grew from fifteen cases to twenty two cases. Six factual, five cross modal, seven analytical, four conversational including a three turn case.
  • IRAC compliance is now intent conditional. Chronology and general queries are scored as not applicable rather than zero, which is fairer to the agent.
  • A small human spot check log at eval_results/human_spot_check.md validates the LLM judge against five sampled cases. Five of five exact agreement on this dataset.

The Big Picture

Before diving into individual files, look at the system end to end. Two things happen offline once. Three things happen on every user message. A fourth thing happens on demand during evaluation.

The offline phase produces a single ChromaDB collection on disk. Every chunk in that collection carries metadata that says which week it belongs to, which document type it came from, and where the source file lives. From that point forward, the agent never touches the original PDFs or images at runtime. All retrieval is metadata aware.

The online phase is a single LangGraph invocation. The router reads the query plus prior chat turns, classifies the intent, and may pull a week filter out of the wording. Retrieval over fetches via MMR, then a cross encoder reranker re scores each candidate against the original query and truncates to the top eight text chunks and four slides. After retrieval the graph forks based on intent. The forks merge at synthesis, which always runs and grounds its answer against the retrieved sources. After synthesis a verification node extracts every case citation in the final answer, checks the retrieved chunks for support, and asks synthesis to rewrite if any citation is unsupported.

Knowledge Base and Indexing

The knowledge base is the foundation of every retrieval call. It is built once by src/indexing/build_index.py and then read at every query.

The indexer walks every week_* directory in order, then walks each subdirectory inside. Two ingestion routines exist. _ingest_pdfs handles text. _ingest_slides handles images.

Text ingestion

Every PDF is opened with PyMuPDF. The pages are joined with a double newline so the chunker can use paragraph boundaries. The RecursiveCharacterTextSplitter from LangChain then breaks the text into 1500 character pieces with 300 character overlap. Why such a large chunk. Legal text is dense. A holding can take half a page. Smaller chunks split sentences across embeddings and retrieval pulls the wrong slices. The 300 character overlap protects against that exact failure for paragraph borders.

splitter = RecursiveCharacterTextSplitter(
    chunk_size=CHUNK_SIZE,        # 1500
    chunk_overlap=CHUNK_OVERLAP,  # 300
    separators=["\n\n", "\n", ". ", " ", ""],
)

Embedding

Embeddings come from ZeroEntropy's zembed-1 via src/embeddings.py. The wrapper class ZeroEntropyEmbeddings follows LangChain's Embeddings contract. It exposes embed_documents for the indexer and embed_query for the retrieval node.

The important detail is the input_type parameter. ZeroEntropy is asymmetric. Document embeddings and query embeddings are produced in different modes so the model can specialise its representation for each role. The wrapper handles that automatically based on which method you call. A 2560 dimensional vector is returned in either case.

def embed_documents(self, texts):
    return self._embed(texts, input_type="document")

def embed_query(self, text):
    return self._embed([text], input_type="query")[0]

The single collection design

One ChromaDB collection (property_law_kb) holds everything. Text and slide chunks live side by side. Metadata filters distinguish them. This is intentional. It keeps retrieval simple. Filter rules build dictionaries that ChromaDB applies natively, so the same MMR call covers both modalities or just one.

The filter builder, with the three cases it handles 
def _build_filter(week=None, doc_types=None):
    conditions = []
    if week:
        conditions.append({"week": {"$eq": week}})
    if doc_types and len(doc_types) == 1:
        conditions.append({"type": {"$eq": doc_types[0]}})
    elif doc_types and len(doc_types) > 1:
        conditions.append({"type": {"$in": doc_types}})
    if not conditions:
        return None
    if len(conditions) == 1:
        return conditions[0]
    return {"$and": conditions}

Three return shapes. None when no constraints were given, a single condition, or an explicit $and when both week and type need to apply.

Layered Retrieval Pipeline

Retrieval in ashGPT is not a single similarity search. It is four layered stages, each responsible for a different failure mode. The whole stack lives in src/agent/tools.py with the cross encoder isolated in src/agent/reranker.py. The retrieval node calls retrieve_all, which fans out to retrieve_texts and retrieve_slides. Both follow the same four stage pattern with different doc type filters.

Returns 16 text or 8 slide candidates"] MMR --> META["Metadata filter

Stage 1. Asymmetric query embedding

The user's query is embedded with zembed-1 in query mode. Because the embedder is asymmetric, the resulting vector is tuned for "short question" representation. A document chunk indexed earlier was embedded in document mode, tuned for "long passage". The dot product between query mode and document mode is what the model is trained on, which gives more meaningful retrieval than a symmetric embedder would for the same content.

Stage 2. Maximal Marginal Relevance over fetch

Plain similarity returns near duplicates. Legal text has a lot of those. The same paragraph from a reading might appear three times because two tutorials and a slide all paraphrase it. MMR re ranks the top fetch_k candidates so that each new pick is rewarded for being relevant and penalised for being too similar to picks already in the result. Lambda 0.5 splits the trade off evenly.

Crucially, MMR is configured to over fetch. It does not return the top eight directly. It returns sixteen text candidates and eight slide candidates. The over fetch creates a diverse candidate pool that the cross encoder can then re score in stage four.

def _search(store, query, k, where_filter, strategy=None):
    strat = strategy or RETRIEVAL_STRATEGY  # "mmr"
    if strat == "mmr":
        return store.max_marginal_relevance_search(
            query, k=k, fetch_k=MMR_FETCH_K,
            lambda_mult=MMR_LAMBDA, filter=where_filter,
        )
    return store.similarity_search(query, k=k, filter=where_filter)

Stage 3. Metadata filtering

The router can pull a week filter out of the user's wording. The Streamlit sidebar can also pre set one, and that overrides whatever the model thinks. The filter is applied at query time as a ChromaDB where clause. The doc type filter is hardcoded inside retrieve_texts (reading, tutorial, supplementary) and retrieve_slides (lecture_slide). That separation is what lets the agent pull both modalities into different state slots.

Stage 4. Cross encoder reranker

The cross encoder is a small pretrained model (ms-marco-MiniLM-L-6-v2) that scores each query and chunk pair directly, rather than relying on the dot product of pre computed embeddings. It is roughly 80 MB on first download and runs locally. The retrieval helper passes the MMR pool into get_reranker().rerank(query, docs, top_k=k) which returns the same documents sorted by cross encoder score, truncated to the originally requested k.

Why both MMR and a cross encoder. MMR enforces diversity, which the IRAC and chronology nodes need. A cross encoder ranks pure relevance, which lifts the most useful chunk to the top. Doing them in sequence gets the best of both. The diversity comes first, so the candidate pool spans different sources. The relevance comes second, so the top k inside that diverse pool is whatever scores highest against the actual question.

def _maybe_rerank(query, docs, top_k, use_reranker):
    if not use_reranker:
        return docs[:top_k]
    try:
        from src.agent.reranker import get_reranker
        reranker = get_reranker()
        return reranker.rerank(query, docs, top_k=top_k)
    except Exception as e:
        log.warning("Reranker unavailable (%s); falling back to MMR order", e)
        return docs[:top_k]

The eval harness can disable the reranker by passing use_reranker=False through the agent state. That is exactly how the no reranker ablation is implemented. Same MMR pool, no re scoring, the original MMR order is truncated to top eight or top four directly.

The packed retrieval query for follow ups

On a follow up turn the new student message alone is often too short to retrieve anything useful. build_retrieval_query in chat_memory.py prepends a short excerpt of the previous tutor reply, so the embedding picks up the topic the student is referring to. The trick is keeping that excerpt short. Dumping a full prior IRAC into the embedding washes out the new question. The cap is CHAT_HISTORY_MAX_ASSISTANT_TAIL_CHARS, set to 1200 characters in config.py.

The Cross Encoder Reranker

src/agent/reranker.py wraps a small pretrained cross encoder. Two parts. A module level singleton accessor and the wrapper class itself.

The lazy singleton

Loading a sentence transformer model takes a few seconds and consumes about 80 MB of memory once it is on disk. The agent does not need that cost on import. get_reranker creates the wrapper instance the first time it is called and caches it in a module level variable. Every subsequent call reuses it.

_singleton = None

def get_reranker():
    global _singleton
    if _singleton is None:
        _singleton = CrossEncoderReranker(model_name=RERANKER_MODEL)
    return _singleton

The wrapper class

The class itself defers the heavy work even further. The model is only built on the first call to rerank. Importing sentence_transformers lives inside _ensure_loaded so the import cost is also lazy. If the package is missing, the helper raises a clear ImportError that names the install command.

class CrossEncoderReranker:
    def __init__(self, model_name=RERANKER_MODEL):
        self.model_name = model_name
        self._model = None

    def _ensure_loaded(self):
        if self._model is None:
            try:
                from sentence_transformers import CrossEncoder
            except ImportError as e:
                raise ImportError(
                    "sentence-transformers is required for CrossEncoderReranker. "
                    "Install it with `pip install sentence-transformers>=3.0.0`."
                ) from e
            log.info("Loading cross-encoder reranker: %s", self.model_name)
            self._model = CrossEncoder(self.model_name)

The rerank call

The actual scoring is a single call to self._model.predict. The input is a list of (query, content) string pairs. The output is a list of float scores aligned with the input pairs. The helper zips documents with scores, sorts descending, and truncates to top_k. Empty input or zero top_k short circuits to an empty list.

def rerank(self, query, documents, top_k):
    if not documents:
        return []
    if top_k <= 0:
        return []
    self._ensure_loaded()
    pairs = [(query, doc.get("content", "")) for doc in documents]
    try:
        scores = self._model.predict(pairs)
    except Exception as e:
        log.warning("Reranker failed (%s); returning original order", e)
        return documents[:top_k]
    scored = list(zip(documents, scores))
    scored.sort(key=lambda x: float(x[1]), reverse=True)
    return [doc for doc, _ in scored[:top_k]]

Code link. The defensive try except mirrors the shape used in llm.py. If predict raises for any reason, retrieval still returns the first top_k documents in MMR order. Failure mode degrades to "agent without reranker", never to "no retrieval".

Why a cross encoder rather than a bi encoder

A bi encoder embeds the query and the document independently and compares with a dot product. ZeroEntropy is a bi encoder, so the initial Chroma search is bi encoder ranking. A cross encoder takes the query and the document together as a single input and produces a relevance score. That direct interaction is more accurate than dot product comparison but cannot scale to millions of documents because it requires one forward pass per pair. The hybrid pattern here is exactly the right shape. Use the bi encoder to fetch a small candidate pool with MMR diversity, then use the cross encoder to re score that small pool. Twenty four pairs per query for retrieval, eight for slides, well within budget for a study tool.

Reranker contribution from the eval suite. Average context precision rises from 0.59 without the reranker to 0.66 with it. The reranker improves precision on 11 of 22 queries, hurts precision on 3 queries, and ties on 8 queries. MRR moves from 0.93 to 0.97. NDCG at K is essentially flat at 0.94 either way. The principal effect is precision, not ranking order, which is consistent with the design role of the layer.

Why total equals false

Total false means every key is optional. A new state can be passed in with just a query and the graph still runs. Every node only writes the keys it owns. This pattern keeps node code tight. A node returns {"intent": "ratio", "node_trace": [...]} rather than rebuilding the whole state.

The reranker override field

use_reranker is a state level escape hatch. It defaults to None, in which case retrieval falls back to the global USE_RERANKER from config. The graph itself lives in src/agent/graph.py. The construction is short, but every line matters because of how LangGraph compiles edges.

How the conditional edges work

LangGraph supports two kinds of edges. A static edge always points to the same next node. A conditional edge invokes a small Python function that reads the current state and returns the name of the next node. ashGPT uses two conditional edges. The first picks the path after retrieval based on the router's intent. The second handles the chained case where summary needs both reasoning nodes.

Static edges look obvious but have a subtle role. Each one is a contract. chronology -> synthesis is always taken. synthesis -> verification is always taken when verification is enabled. verification -> END closes every path. The mix of static and conditional edges keeps the graph readable.

The cached singleton

Compiling a LangGraph is not free. It builds an internal representation, validates the edges, and hashes the node functions. The repository compiles once and caches via get_graph. run_query calls that cache. This means the first message of a session pays the compile cost and every subsequent message is fast.

The Six Cognitive Nodes

Every node in src/agent/nodes.py is a pure function. State in. Partial state out. The router classifies. Retrieval reads from Chroma and runs the layered pipeline. Ratio extractor and chronology generator each do one specialised reasoning task. Synthesis grounds the final answer. Verification fact checks every citation in the final answer. The same shape repeats six times.

Router intent classifier and week filter extractor

Reads. query, chat_history, an optional pre set week_filter. Writes. intent, week_filter, node_trace.

The router is the most important node for hypothesis testing. If routing is wrong, ratio cases never run the ratio path and chronology cases never get a diagram. Its system prompt is therefore very explicit about the four classes and the decision rule that disambiguates summary from chronology.

The router model is gemini-3.1-flash-lite-preview. It is the cheapest model in the pipeline because the task is just classification. The output is parsed by stripping any optional code fences and calling json.loads. If parsing fails the router defaults to summary so the user still gets a complete answer.

Week filter precedence

If the Streamlit sidebar already passed a week_filter, that value wins. The router only sets week_filter when the caller did not. This gives the user explicit control.

Ratio Extractor the IRAC engine

Reads. query, retrieved_texts, retrieved_slides, chat_history. Writes. ratio_decidendi, irac_analysis, node_trace.

This node implements one half of the design hypothesis. It separates the binding rule from the rest of the case material. Its system prompt is two paragraphs of constraint. Pull a single sentence ratio. Produce four labelled sections. Cite only what is in the source material.

Notice the careful split inside RATIO_SYSTEM. Facts are constrained. Explanations are encouraged. Without that distinction, the model either invents to fill gaps or refuses to clarify. The wording was chosen so a tutor style answer is allowed without inviting hallucination.

Chronology Generator the diagram engine

Reads. query, retrieved_texts, retrieved_slides, chat_history. Writes. mermaid_diagram, chronology_summary, node_trace.

This is the other half of the hypothesis. The diagram is the visible artefact that proves the agent reasoned about chronology as its own task. The system prompt insists on Mermaid graph TD, camelCase IDs, short labels, edge labels for detail, and 5 to 12 nodes.

Parsing the response

The node uses two regular expressions. One captures the fenced Mermaid block. The other captures the timeline summary that follows. Both are stored separately in state. Synthesis later inserts the Mermaid block verbatim into the final answer, and the timeline summary turns into prose.

Why a small model is fine here

The chronology task is structurally constrained. There is no need for heavy reasoning. The default model is gemini-3-flash-preview. It is fast and cheap, and the system prompt does most of the structural work.

Synthesis the grounded final answer

Reads. Everything. Writes. final_answer, node_trace.

The synthesis node produces what the student actually reads. It receives the original query, the intent, the retrieved sources, and any artefacts from the upstream reasoning nodes. Its system prompt enforces three things. Ground every fact in the primary evidence. Treat IRAC and chronology as derived analysis that still needs verification. Reproduce any Mermaid block verbatim.

The PRIMARY EVIDENCE vs DERIVED ANALYSIS split

The user prompt builds two clearly labelled blocks. PRIMARY EVIDENCE contains the raw retrieved chunks. DERIVED ANALYSIS contains the IRAC, the ratio sentence, the Mermaid block, and the timeline summary. The model is told to verify every claim from PRIMARY EVIDENCE, even if it appears in DERIVED ANALYSIS. This protects the system from a hallucination that snuck into ratio extraction propagating into the final answer.

Why the Mermaid block is copied verbatim

If the synthesis model paraphrased the chart into bullet points, the Streamlit UI could not render it. The whole point of the chronology path would be lost in the last step. The system prompt and the user prompt both repeat the rule, and a follow up sentence makes it clear. The UI renders the diagram, do not convert it to bullet points.

The frontend reinforces this. If the synthesis output omits the Mermaid block but the state has one, the UI appends it before storing the message. That guardrail is what guarantees diagrams survive multi turn replay.

Verification citation regex and rewrite

Reads. final_answer, retrieved_texts, retrieved_slides, query. Writes. final_answer (possibly rewritten), verification_report, node_trace.

The verification node is the post synthesis safety net. It extracts every plausible X v Y case citation from the final answer, normalises each one, and checks for full string or two token presence in the concatenated retrieved chunk text. If any citation is unsupported, it asks the synthesis model to rewrite the answer with that claim removed or hedged, while preserving every other heading and any Mermaid block.

See the next section for the full anatomy of the citation extractor and the rewrite branch. The headline result from the eval suite is that the node fired on all 22 of 22 cases (the citation regex always runs) and the regex extracted zero unsupported citations across the suite, which is consistent with the synthesis prompt's strict grounding rules. The node remains active as a low cost safety net that intercepts hallucinated case citations the synthesis model might otherwise import from parametric memory.

The Verification Node

The verification node is split between two files. src/agent/verification.py contains the regex based citation extractor and a substring check against retrieved chunk text. src/agent/nodes.py contains the node itself, which orchestrates extraction, the unsupported list, and the rewrite branch.

Why a verification node at all

Synthesis is told strictly to ground every factual claim, but LLMs occasionally drift. A familiar Property Law case from the model's parametric memory might surface in the final answer even if the retrieval pool does not contain that case. The verification node catches that drift cheaply. It does not call any LLM by default. It only invokes the synthesis model if a citation needs to be removed or hedged.

The citation regex

The extractor is intentionally permissive on formatting (italics, underscores, plain text) but constrained on shape (X v Y or X vs Y). It also tolerates multi word case names like Buckinghamshire County Council v Moran by allowing up to four extra capitalised tokens on the left side, with stop words like "of", "the", "and" permitted between them.

_CASE_NAME_PART = (
    r"[A-Z][A-Za-z'’-]+"
    r"(?:\s+(?:of|the|and|de|du|von|van)\s+[A-Z][A-Za-z'’-]+)?"
    r"(?:\s+[A-Z][A-Za-z'’-]+){0,4}"
)

_CASE_PATTERN = re.compile(
    r"(?P[*_]{1,2})?"
    rf"(?P{_CASE_NAME_PART})"
    r"\s+(?:v|vs)\.?\s+"
    rf"(?P{_CASE_NAME_PART})"
    r"(?P=wrap)?",
    re.UNICODE,
)

Trimming sentence initial words

Without the trim, the multi word left side of the regex absorbs sentence initial English words like "In", "The", or "However". The extractor strips those leading stopwords before normalising the citation. The _LEFT_TRIM_PREFIXES set covers articles, prepositions, conjunctions, and a few legal connectors like "Section" and "See".

def _trim_leading_stopwords(left):
    parts = left.split()
    while parts and parts[0].lower() in _LEFT_TRIM_PREFIXES:
        parts.pop(0)
    return " ".join(parts)

Checking each citation against the sources

For each candidate citation the helper normalises the text (lowercase, strip italic markers, collapse whitespace) and asks two questions in order. Does the full X v Y string appear in the concatenated source text. If not, do both party tokens (the full left side and the full right side) appear individually. The two token fallback is what catches citations where the source quotes the case as "In Smith and Jones the court held..." without the v separator.

def case_appears_in_sources(case, sources_text):
    norm_sources = _normalise(sources_text)
    norm_case = _normalise(case)
    if norm_case in norm_sources:
        return True
    parts = [p.strip() for p in re.split(r"\sv\s", norm_case) if p.strip()]
    if len(parts) != 2:
        return False
    left, right = parts
    return (left in norm_sources) and (right in norm_sources)

The node itself

verification_node in nodes.py stitches the helpers together. It always logs into verification_report, regardless of whether the rewrite branch fires. If find_unsupported_cases returns an empty list, the node short circuits with the report and appends to the trace. If it returns one or more citations, the node builds a rewrite prompt and calls llm_call with the synthesis model and a dedicated rewrite system instruction.

def verification_node(state):
    answer = state.get("final_answer", "") or ""
    sources_text = _collect_sources_text(state)
    unsupported = find_unsupported_cases(answer, sources_text)

    report = {"unsupported_claims": unsupported, "rewrites_applied": False}

    if not unsupported:
        return {
            "verification_report": report,
            "node_trace": _append_trace(state, "verification"),

    rewrite_prompt = (
        f"STUDENT QUESTION: {state.get('query', '')}\n\n"
        f"RETRIEVED SOURCE MATERIAL:\n{_format_context(state)}\n\n"
        f"DRAFT ANSWER:\n{answer}\n\n"
        f"UNSUPPORTED CITATIONS (must be removed or hedged):\n"
        + "\n".join(f"- {c}" for c in unsupported)
        + "\n\nRewrite the answer per the instructions. Return ONLY the rewritten answer."
    )

    try:
        rewritten = llm_call(
            rewrite_prompt,
            model=SYNTHESIS_MODEL,
            system_instruction=VERIFICATION_REWRITE_SYSTEM,
        )

            report["rewrites_applied"] = True
            return {
                "final_answer": rewritten,
                "verification_report": report,
                "node_trace": _append_trace(state, "verification"),
            }
    except Exception as e:
        log.warning("VerificationNode: rewrite failed (%s); keeping draft", e)

    return {
        "verification_report": report,
        "node_trace": _append_trace(state, "verification"),
    }

Multi Provider LLM Layer

One module, src/llm.py, hides the differences between Google, OpenAI, and Anthropic. Every node calls a single function, llm_call(prompt, model, system_instruction, temperature). The dispatcher reads the model name prefix and forwards to the right SDK.

def llm_call(prompt, model, system_instruction=None, temperature=0.2, reasoning_effort=None):
    if model.startswith("gemini"):
        return _call_gemini(prompt, model, system_instruction, temperature)
    elif model.startswith("gpt"):
        effort = reasoning_effort or _REASONING_EFFORT.get(model)
        return _call_openai(prompt, model, system_instruction, temperature, effort)
    elif model.startswith("claude"):
        return _call_anthropic(prompt, model, system_instruction, temperature)
    else:
        raise ValueError(f"Unknown model prefix: {model!r}")

Lazy clients

The Gemini, OpenAI, and Anthropic SDK clients are only instantiated when first needed. If a configuration uses only Gemini and OpenAI, the Anthropic SDK is never imported. This keeps cold start fast and the dependency footprint minimal during runtime even though the requirements file installs all three.

Per provider quirks

ProviderSpecial handling
GeminiCalls generate_content with a GenerateContentConfig that includes the system instruction. Token usage comes from usage_metadata.
OpenAIUses the new responses.create API. System instructions are passed as the first message. Reasoning models accept a reasoning_effort hint, which is auto stripped on retry if the model rejects it.

Token tracking

Every successful call records input and output token counts on a global TokenUsage object. Nodes never need to think about tracking. The eval harness calls reset_token_usage before each case and get_token_usage().summary() after, which is how the per case totals end up in the eval JSON.

Why temperature equals 0.2 by default. Reasoning is meant to be deterministic enough for a study aid, but slight diversity helps the model produce natural prose. Judges always use temperature 0 because they should not vary across replays.

Multi Turn Chat Memory

src/agent/chat_memory.py handles three concerns. Sanitising raw history. Formatting history for LLM prompts. Building a focused retrieval query when the user is following up.

Sanitising history

def prepare_chat_history_for_run(raw):
    if not raw:
        return []
    out = []
    for m in raw:
        role = m.get("role", "")
        content = (m.get("content") or "").strip()
        if role not in ("user", "assistant") or not content:
            continue
        cap = CHAT_HISTORY_MAX_CHARS_PER_MESSAGE  # 3500
        if len(content) > cap:
            content = content[:cap] + "\n… [truncated]"
        out.append({"role": role, "content": content})
    if len(out) > CHAT_HISTORY_MAX_MESSAGES:    # 24
        out = out[-CHAT_HISTORY_MAX_MESSAGES:]
    return out

Empty messages and unknown roles are dropped. Any single message longer than 3500 characters is hard truncated. The whole history is capped at the most recent 24 messages. These limits live in config.py so they can be tuned without code changes.

Formatting history

For LLM prompts the history is rendered as a numbered transcript with the labels Student and Tutor. This is what the router, ratio extractor, chronology generator, and synthesis node all see when there are prior turns.

def format_transcript_for_llm(messages):
    parts = []
    for i, m in enumerate(messages, start=1):
        label = "Student" if m["role"] == "user" else "Tutor"
        parts.append(f"[{i}] {label}: {m['content']}")
    return "\n\n".join(parts)

Packed retrieval query

This is the small detail that makes follow ups work without overwhelming the embedding. When the latest history entry is a tutor reply, a short excerpt of that reply is prepended to the new student question before embedding.

def build_retrieval_query(current_query, messages):
    if not messages:
        return current_query
    for i in range(len(messages) - 1, -1, -1):
        if messages[i]["role"] != "assistant":
            continue
        excerpt = messages[i]["content"][:CHAT_HISTORY_MAX_ASSISTANT_TAIL_CHARS].strip()
        if not excerpt:
            break
        return (
            "[Conversation follow-up — retrieve material relevant to both the prior "
            "answer and the new question.]\n\n"
            f"Prior tutor answer (excerpt):\n{excerpt}\n\n"
            f"New student question:\n{current_query}"
        )
    return current_query

If the user asks "give me an exam tip tailored to that ratio" with no excerpt the embedding has nothing to anchor on. With a short excerpt the embedding moves toward the right region of the vector space and retrieval pulls back useful chunks.

The Streamlit Frontend

app.py is the chat interface. It is short on purpose. Most of the heavy lifting happens in the agent. The frontend's responsibilities are session memory, message rendering, and the trace banner.

Session memory

Streamlit's st.session_state holds a messages list. Each message is a dict with a role and content, exactly the shape the agent expects. When the user sends a new prompt, the prior messages become chat_history for that single run_query call.

render_message

This helper is the only place that decides how to display content. It splits text on the Mermaid fence regex. Plain segments go to st.markdown. Mermaid segments go to st_mermaid, which renders an interactive SVG diagram.

def render_message(content):
    parts = re.split(r"(```mermaid\n.*?```)", content, flags=re.DOTALL)
    for part in parts:
        m = re.fullmatch(r"```mermaid\n(.*?)```", part, flags=re.DOTALL)
        if m:
            st_mermaid(m.group(1).strip(), height="auto")
        elif part.strip():
            st.markdown(part)

This same helper renders both the live response and any historic message during chat replay, which is exactly why diagrams keep working as the conversation grows.

After a response renders, three metric boxes appear above it. Detected intent. Total latency in seconds. The node trace as a single arrow joined string. Underneath, three optional expanders show the raw Mermaid code, the full IRAC analysis, and the source list. This is what makes the system inspectable while you study with it.

The sidebar exposes a week filter dropdown and a clear conversation button. Selecting a week from the dropdown sets week_filter on the next call and overrides anything the router would extract. Clear conversation wipes st.session_state.messages and reruns the page.

LLM as Judge Framework

Three judges live inside the eval harness. Groundedness is the rich one. Relevancy and context precision are simpler.

Groundedness. Two stage cross provider

Stage 1. Draft

Model. JUDGE_DRAFT_MODEL. Currently Gemini. Receives the question, the source bundle, and the answer. Returns {"score": 1..5, "reasoning": "..."}. The rubric tells the judge to penalise invented facts but accept paraphrase and plain English explanation.

Stage 2. Critique

Model. JUDGE_CRITIQUE_MODEL. Currently OpenAI. Receives the same materials plus the draft score and reasoning. Acts as a senior reviewer. Returns {"score": 1..5, "reasoning": "...", "agreed_with_draft": bool}. The critique score is the final number, with a fallback to the draft on parse failure.

Why two stages with two providers. To reduce same model self evaluation bias. If the agent is built mostly on OpenAI, a judge that is also OpenAI inherits the same blind spots. Cross provider judging breaks that loop. Stage two acts as a double check on stage one rather than as a fresh independent grader, which keeps cost and noise low.

Relevancy. Single stage

One call. Question and answer only, no source. The judge scores 1 to 5. Asks whether the answer addresses the question. This catches a high quality but off topic response that grounded scoring would miss.

Context precision. Per chunk binary

For each retrieved chunk the judge sees the question and the chunk content (truncated to 1500 characters), and returns {"relevant": true|false}. Then ranking metrics are computed from the verdict list in production order.

def _ranking_metrics_from_binary_verdicts(verdicts):
    k = len(verdicts)
    if k == 0:
        return {"mrr": 0.0, "hit_at_k": 0.0, "ndcg_at_k": 0.0, "k": 0}
    mrr = 0.0
    for i, v in enumerate(verdicts):
        if v:
            mrr = 1.0 / (i + 1); break
    hit_at_k = 1.0 if any(verdicts) else 0.0
    dcg = sum((1.0 if v else 0.0) / math.log2(i + 2) for i, v in enumerate(verdicts))
    rel_count = int(sum(1 for v in verdicts if v))
    idcg = sum(1.0 / math.log2(i + 2) for i in range(min(rel_count, k)))
    ndcg_at_k = (dcg / idcg) if idcg > 0 else 0.0
    return {"mrr": round(mrr, 3), "hit_at_k": hit_at_k,
            "ndcg_at_k": round(ndcg_at_k, 3), "k": k}

Optional pool eval

--retrieval-pool-eval retrieves a larger candidate pool (configured by EVAL_RETRIEVAL_POOL_K_*). Every chunk in the pool gets judged once. The production sized slice keeps its precision and ranking metrics. A new metric, recall vs pool, divides relevant in production by relevant anywhere in the pool. This is a bounded proxy for recall, since there are no human qrels for a corpus wide ground truth, and the report should treat it as such.

Structural checks

These do not call any model. They run rule based regular expressions.

  • Mermaid validity. Five binary checks. Has graph header. Has nodes. Has edges. At least three nodes. No empty labels. Score is the fraction passed.
  • IRAC compliance. Four binary checks for Issue, Rule, Application, Conclusion using case insensitive regex. The check is intent conditional. If the router classified the query as chronology or general, the helper returns {"applicable": False, "score": None} instead of zero. The summary excludes None scores from the average and reports irac_applicable_count alongside.
_IRAC_NA_INTENTS = {"chronology", "general"}

def check_irac_compliance(irac_text, intent=None):
    if intent in _IRAC_NA_INTENTS:
        return {"compliant": None, "score": None, "applicable": False, ...}
    if not irac_text or not irac_text.strip():
        return {"compliant": False, "score": 0.0, "applicable": True, ...}
    components = {
        "issue": bool(re.search(r"\b(issue|legal question)\b", irac_text, re.I)),
        "rule": bool(re.search(r"\b(rule|ratio decidendi|legal (rule|principle|test))\b", irac_text, re.I)),
        "application": bool(re.search(r"\bapplicat", irac_text, re.I)),
        "conclusion": bool(re.search(r"\bconclus", irac_text, re.I)),
    }
    ...

This is the conditional IRAC fix. Without it, the agent routed correctly to chronology for a "show me a flowchart" query and was penalised for not producing IRAC, even though IRAC was never asked for. The conditional check is fairer and is part of what the report explains.

Systemic weaknesses

Future work signposted in the report

  1. Hybrid retrieval. BM25 plus dense embeddings via reciprocal rank fusion would lift exact statute and case name surface forms above topical neighbours and address the 0.52 factual precision.
  2. Latency reduction. Streaming node execution and smaller local models for the router and chronology could compress end to end latency toward 10 to 12 seconds while preserving routing accuracy.
  3. Broader verification. The current verification node only checks case citations. Statute and slide identifier checks would extend the safety net to the families where the regex does not yet fire.

Decisions and Trade Offs

Every non obvious choice in the codebase is here with the reasoning. Each one is phrased as a question an interviewer might actually ask.

Why a multi node graph instead of a single careful prompt

The hypothesis itself is the answer. Cognitive separation against attention dilution. The mega prompt ablation tests this directly with the same retrieval bundle. The agent gained 0.09 groundedness points and 9 percentage points of Mermaid validity over the ablation. The only difference between them is the LangGraph state machine, so those gains attach to node specialisation, not to better evidence or stronger models.

Why MMR plus a cross encoder instead of just MMR or just BM25

Legal text contains many near duplicate paragraphs across readings, slides, and tutorials. Plain similarity returns ten variations of the same passage. MMR fetches a diverse candidate pool. The cross encoder then re scores that pool against the actual question, which lifts the most relevant chunks above the diversity driven candidates. Using both gives you a diverse pool first and pure relevance ranking inside it second. Hybrid BM25 plus dense remains the next future work item to lift exact statute and case name surface forms.

Why a verification node when synthesis is already grounded

Synthesis is told strictly to ground every claim. Models drift anyway. The verification node is a cheap regex sweep over the final answer that catches case citations missing from the retrieved chunks. Across 22 cases it fired 22 times and intercepted zero unsupported citations, which means the synthesis prompt is doing its job today, but the safety net stays in place for the day a model regression sneaks an out of context case into a final answer.

Why 1500 character chunks with 300 character overlap

Legal text is dense. A single ratio holding can span half a page. 1500 characters preserves the typical paragraph plus a sentence of context on either side. The 300 character overlap protects against splits that fall in the middle of a key clause.

Why over fetch 16 text and 8 slide candidates instead of 8 and 4 directly

The cross encoder needs a candidate pool to rerank. Without over fetch the reranker has nothing to choose from. The over fetch creates a diverse pool from MMR, then the cross encoder re scores and truncates to the production sized 8 plus 4. This is what lifts context precision from 0.59 to 0.66 in the eval.

Why one Chroma collection instead of one per modality

Two reasons. First, retrieval has to sometimes return both modalities for the same query, which is simpler with metadata filters than with two collections and a merge step. Second, the multimodal eval treats slides and texts on the same footing, and using one collection means the same MMR pass and the same embedding model apply to both.

Why describe slides with a VLM at index time instead of querying images at runtime

Indexing once is cheap relative to querying every time. The VLM produces a verbose description that captures every visible piece of text, every diagram, and every case name verbatim. Cross modal retrieval hit both_modalities=True on every cross modal case, which validates this design.

Why summary fires both reasoning nodes

A query like "explain X and give me the sequence of events" needs both an IRAC and a Mermaid diagram. Routing to chronology alone produces a bare diagram with no analysis. Routing to ratio alone produces analysis with no timeline. Summary is the path that fires both.

Why per node model assignments

Each node has a different cognitive complexity. Routing is classification and tolerates a cheap model. Chronology is structurally constrained and tolerates a small model. Ratio extraction needs structured legal reasoning and uses a mid tier model. Synthesis needs the strongest grounded reasoning. Per node assignments mean cheap models do cheap work.

Why three providers

To break same model self evaluation bias. The agent uses Gemini in some nodes and OpenAI in others. The judge uses Gemini for the draft and OpenAI for the critique.

Why a two stage groundedness judge

A single LLM judge inherits all the blind spots of its provider. Two stages with two providers act as a check and balance. The eval JSON keeps both scores so the disagreement rate can be inspected. The five case human spot check shows zero disagreement on this dataset, which is a sanity check on the judge framework.

Why intent conditional IRAC scoring

Without it, the agent routed correctly to chronology on a "show me a flowchart" query and was penalised for not producing an IRAC, even though IRAC was never asked for. The conditional helper returns "not applicable" for chronology and general intents, the summary excludes those scores from the average, and the report quotes the applicable case count alongside the average for transparency.

Why total equals false on AgentState

It lets every key be optional at graph entry. A new state can be passed in with just a query. Each node only writes the keys it produces.

Why a packed retrieval query for follow ups

Embedding the entire prior tutor reply washes out the new student question. A 1200 character excerpt of the prior reply gives the embedding enough topic anchor without dominating the vector. After the embedding goes through MMR the cross encoder re scores against the packed query, which keeps the final pool tight on the actual follow up topic.

Why the synthesis prompt mandates a verbatim Mermaid block

Because the model would otherwise paraphrase the diagram into bullet points, and the Streamlit UI cannot render bullet points as a diagram. Mermaid only renders if the exact fenced block reaches the rendered message. Three guards: the system prompt, the user prompt, and the frontend append guardrail.

Why default to summary on router parse failure

Summary fires both reasoning nodes. If we default to general the user gets the simplest possible answer. If we default to summary the user gets the most complete answer.

Why temperature 0.2 for nodes and temperature 0 for judges

Reasoning needs slight diversity for prose flow. Judges should be reproducible across runs.

Why per node latency tracking

Because the design hypothesis claims a cost. The 0.09 groundedness gain costs the user 14 extra seconds versus the mega prompt. Per node latencies make that cost concrete. Chronology at 8.6s and ratio extractor at 6.6s dominate.

Why recall vs pool is a proxy and not real recall

Real recall needs a fully labelled relevant set across the whole corpus. The eval has no human qrels. Recall vs pool uses the relevant chunks anywhere in the enlarged retrieval pool as a stand in.

Why the no reranker configuration is a separate full run instead of a metadata flag

Because the eval needs to score the same metrics on the same case with the cross encoder disabled. Re running the agent with use_reranker=False via the state override produces a clean fourth column in the results that isolates the reranker's contribution.

Interview Cheat Sheet

Crisp question and answer pairs. Each one is the shortest defensible answer in your own voice.

Q1. What is the project

A multimodal LangGraph agent that helps a law student study Australian property law. Six cognitive nodes including a verification stage. Dual index over PDFs and VLM described slides. Layered retrieval with MMR plus cross encoder. Four configurations evaluated head to head.

Q2. What is the hypothesis

Cognitive separation versus attention dilution. A multi node LangGraph state machine that separates ratio decidendi extraction from chronological fact extraction will produce more accurate causal analyses than a single consolidated mega prompt baseline that receives identical retrieved context.

Q3. What is the result

Full agent at 4.86 groundedness, mega prompt at 4.77, no reranker variant at 4.86, baseline at 3.41, on the same 22 cases. The 0.09 groundedness gain over the mega prompt is attributable to cognitive separation. The reranker lifts context precision from 0.59 to 0.66.

Q4. Walk me through one query end to end

User message hits Streamlit. run_query builds a state. Router classifies intent. Retrieval over fetches via MMR and the cross encoder reranks. The graph forks based on intent. Reasoning nodes write IRAC or Mermaid. Synthesis grounds everything in primary evidence. Verification fact checks every case citation. The UI renders inline.

Q5. Why LangGraph and not LangChain

Because the design needs conditional branching after retrieval. Standard chains are linear. A state machine with conditional edges expresses the four routes naturally and lets per node outputs be inspected and ablated independently.

Q6. Why ZeroEntropy zembed-1

Asymmetric embeddings tuned for retrieval. Documents and queries are embedded in different modes. 2560 dimensions. Wrapped by hand because LangChain has no built in client.

Q7. Why MMR with lambda 0.5 and over fetch

MMR balances relevance and diversity. The over fetch creates a 16 text or 8 slide candidate pool that the cross encoder re scores. Diversity comes first via MMR, then pure relevance ranking via the cross encoder.

Q8. Why a cross encoder reranker on top of MMR

A bi encoder embeds query and document independently. A cross encoder takes both as a single input and produces a relevance score directly. More accurate but only feasible on a small candidate pool. The hybrid pattern combines bi encoder recall (MMR) with cross encoder precision (rerank).

Q9. How do slides become retrievable by text

Gemini 2.5 Pro reads each slide image at index time and produces a verbose text description. The description is embedded with the same model as the readings. A text query naturally pulls back relevant slides.

Q10. What is the four intent classification

Ratio for binding rule and IRAC. Chronology for diagrams alone. Summary when the user wants both an explanation and a timeline. General for greetings or off topic.

Q11. Why does summary fire both reasoning nodes

Because a "explain X and give me the sequence of events" query needs both an IRAC and a Mermaid diagram. The decision rule in the router pushes any combined query to summary.

Q12. What does the verification node actually do

Extracts every plausible X v Y case citation from the final answer using a permissive regex. Checks each one for full string or two token presence in the concatenated retrieved chunk text. If anything is unsupported, asks the synthesis model to rewrite the answer with that claim removed or hedged. Across 22 eval cases it fired 22 times and intercepted zero unsupported citations.

Q13. Why is verification so fast at 0 seconds

Because the regex check is sub millisecond and the rewrite branch did not trigger across the suite. The expensive call to the synthesis model only happens when a citation needs editing. Otherwise the node is essentially free.

Q14. How is the Mermaid diagram protected from being lost

Three guards. The synthesis system prompt mandates a verbatim copy. The user prompt repeats the rule. The Streamlit frontend appends the Mermaid block to the stored message if synthesis omits it. The verification rewrite system prompt also preserves Mermaid blocks verbatim.

Q15. What does the multi provider dispatcher do

Routes llm_call by model name prefix. gemini- goes to Google. gpt- goes to OpenAI Responses API. claude- goes to Anthropic. Lazy clients. Token tracking.

Q16. How does conversational follow up work

Streamlit holds messages in st.session_state. Each new turn passes prior messages as chat_history. The router uses the transcript to resolve "that case". Retrieval prepends a 1200 char excerpt of the prior tutor reply to the embedding query. The reranker then re scores against the packed query.

Q17. What is the mega prompt ablation

Same retrieval bundle as the agent. One LLM call replaces every reasoning node and synthesis. Produces all three sections in one pass. Used to measure whether multi node decomposition adds anything beyond a well prompted single call.

Q18. Why does the mega prompt score 100% IRAC compliance over all 22 cases but the agent only over 19

Because the agent's IRAC scoring is intent conditional. Chronology and general queries are scored not applicable instead of zero. The mega prompt produces all three sections unconditionally, including for queries that should not have IRAC, which is over triggering.

Q19. What is the no reranker ablation

The full agent run again with use_reranker=False in the agent state. MMR returns the production sized 8 text and 4 slide chunks directly. Same graph, same verification, no cross encoder. Isolates the reranker's contribution.

Q20. What is the two stage groundedness judge

A draft judge from Gemini scores 1 to 5 with reasoning. A critique judge from OpenAI reviews the draft. The critique score is final. Cross provider design reduces same model self evaluation bias.

Q21. How do you know the LLM judge is reliable

The eval harness writes a five case human spot check log. A human fills in their own scores by hand. Across the five sampled cases the human and LLM judge agreed exactly on every case for both groundedness and relevancy. Five of five exact agreement, MAE 0.0.

Q22. What metrics does the eval collect

Groundedness, answer relevancy, context precision @K, MRR, Hit @K, NDCG @K, optional recall vs pool, source diversity, total latency, per node latency, token usage, Mermaid validity, intent conditional IRAC compliance, cross modal modality flags, verification telemetry.

Q23. What is recall vs pool and why isn't it real recall

It divides relevant chunks in the production slice by relevant chunks anywhere in an enlarged retrieval pool. Real recall needs human qrels across the whole corpus. The pool is a bounded proxy.

Q24. What family had the lowest context precision and why

Factual retrieval at 0.52. Statute and named test queries miss the exact surface form of the target inside dense embedding space. The fix is hybrid retrieval with BM25 plus dense via reciprocal rank fusion.

Q25. What is the agent's biggest weakness

Retrieval imprecision, not reasoning error. Average context precision is 0.66 and 15 of 22 queries are below 0.70. Per node reasoning is strong enough to filter noise, but the cost is wasted context tokens and elevated latency.

Q26. What is on the future work list

Hybrid BM25 plus dense retrieval with reciprocal rank fusion. Streaming and smaller local models for the router and chronology to compress latency. Broader verification beyond case citations to statutes and slide identifiers.

Q27. What does node_trace let you do

Verify which path actually ran, debug routing, surface per node latency, and prove in the eval that the right intent took the right route. Verification appears at the end of every trace when enabled.

Q28. What happens if the cross encoder fails

The _maybe_rerank helper wraps the call in try except. On any failure it returns the first top_k documents in MMR order. The graph still runs, retrieval still returns useful chunks, the failure mode degrades to "agent without reranker".

Q29. What happens if Chroma is empty

Retrieval returns empty lists. The synthesis prompt says explicitly that if sources are insufficient the model should reply with a "consult your additional readings" message rather than fabricate.

Q30. What does the verification report look like in state

{"unsupported_claims": [], "rewrites_applied": False} on a clean answer. {"unsupported_claims": ["Smith v Jones"], "rewrites_applied": True} when the rewrite branch fired. Both fields are recorded in the eval JSON and surface in the live counter log line.

Glossary

Ratio decidendi

The binding legal principle the court applied to reach its decision. The single most important sentence in any common law judgment.

IRAC

Issue, Rule, Application, Conclusion. The standard shape used to write up a case in Anglo Australian legal education.

Animus possidendi

Intention to possess. One of the two limbs of adverse possession alongside factual possession.

MMR

Maximal Marginal Relevance. A re ranking method that balances relevance and diversity. ashGPT uses lambda 0.5 over the top 20 candidates with over fetch.

Cross encoder

A model that takes a query and a document together as a single input and produces a relevance score directly. More accurate than bi encoder dot product, only feasible on a small candidate pool.

Bi encoder

A model that embeds the query and the document independently and compares with a dot product. ZeroEntropy is a bi encoder.

Asymmetric embedding

A model that embeds queries and documents in different modes so the vector representations are tuned for retrieval rather than for symmetric similarity.

VLM

Vision Language Model. ashGPT uses Gemini 2.5 Pro to read each slide image and produce a text description that gets embedded.

Mermaid

A text based diagramming syntax. ashGPT generates graph TD flowcharts. The Streamlit UI renders them as interactive SVGs.

LLM as a judge

Using a language model to grade another language model's output. ashGPT uses cross provider judging plus a critique stage to reduce same model bias.

Groundedness

How well an answer's factual claims are supported by the source material it was given.

Recall vs pool

A bounded proxy for recall. Relevant chunks in the production slice divided by relevant chunks anywhere in an enlarged candidate pool, judged once.

Conditional edge

A LangGraph edge whose target depends on a small Python function reading the current state. ashGPT uses two of these.

TypedDict

Python typing primitive used to describe the shape of a dict. ashGPT uses total=False so all keys are optional.

Verification node

Post synthesis fact check. Extracts case citations from the final answer with a regex, checks each one against the retrieved sources, asks the synthesis model to rewrite the answer with unsupported citations removed.

Attention dilution

The failure mode the design hypothesis names. A monolithic prompt asked to produce IRAC plus Mermaid plus synthesis silently trades one structural artefact for another inside one decoder pass.

Over triggering

The mega prompt's failure mode. A prompt that always emits every section regardless of whether the question asked for it. Inflates compliance metrics with output the user did not request.

Cognitive separation

The design hypothesis's named pattern. Splitting structurally distinct outputs into specialised nodes so each one is generated under its own constraints rather than under one consolidated directive.
